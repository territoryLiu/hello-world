# Travel Skill V2 Design

## 背景

当前仓库中的 `travel-skill` 第一阶段已经完成了基础链路：

1. intake 标准化
2. research task 展开
3. source merge / fact extraction / review packet
4. guide model composition
5. render / single-html / zip
6. verify / Playwright 接入

这条链路已经能把旅游请求转换为结构化内容、HTML 页面和分享产物，但仍然存在四个明显问题：

1. 内容分层还不够细。当前更接近“一份完整攻略”，还没有明确区分 `每日行程速览`、`最推荐攻略`、`最全面攻略` 三层产物。
2. 研究数据的复用深度还不够。下一版需要把联网结果按“地点 + 主题”双层归档，方便后续补充、二次生成和多端复用。
3. 页面成品规则还不够稳定。桌面端和手机端虽然都能生成，但尚未升级为“每层攻略都同时输出桌面端与手机端”的稳定合同。
4. 编排层对已安装 skill 的分工还不够清楚。下一版需要把 `web-access`、`frontend-design`、`theme-factory`、`playwright-skill` 的职责边界写入正式规范。

本设计文档面向 `travel-skill` 下一版，定位为“国内多城市、多交通方式、强信息密度”旅行攻略生产规范。延吉 / 长白山是代表性样例，但规范本身不锁死在单一路线。

## 设计目标

`travel-skill` V2 的目标不是只生成一篇旅游攻略，而是生成一套可复用的旅游攻略生产系统。默认需要满足：

1. 统一基于一份研究底座，生成三层内容产物：
   - `每日行程速览`
   - `最推荐攻略`
   - `最全面攻略`
2. 每层内容都同时输出：
   - 桌面端 HTML
   - 手机端 HTML
3. 手机端采用分页式阅读，内容与桌面端一致，只改变阅读组织方式，不删减核心信息。
4. 联网结果按“地点 + 主题”双层 JSON 存储，并同步生成总汇总文件。
5. 高时效信息默认采用“最近可查方案 + 价格区间 + 核对日期”的正文写法，详细班次和明细进入 JSON 与附表。
6. 默认产出可分享的单文件 HTML、入口导航页和 ZIP 打包结果。
7. `travel-skill` 自身作为编排 skill，负责调用其他 skill 和本地脚本，而不是把所有专项能力写死在一个步骤里。

## 非目标

以下内容不属于 V2 的默认强约束：

1. 不要求默认同时产出 2 到 3 套完全不同视觉风格页面。
2. 不要求默认具备公网部署能力，公开 URL 仍属于增强项。
3. 不要求跳过人工确认闸门，全流程保留 `review-gate`。
4. 不要求把未来精确天气、未来精确票价、未来精确营业状态写成确定事实。

## 核心决策

### 1. 产品分层

V2 固定采用三层攻略产物：

1. `daily-overview`
   - 用于快速理解每天怎么走、怎么住、怎么接交通
   - 适合桌面端首页、手机端首页、单文件导航入口
2. `recommended`
   - 给出默认最合适方案
   - 面向大多数读者直接执行
3. `comprehensive`
   - 给出更大的选择空间
   - 收纳更多交通备选、店铺清单、景点信息和来源记录

这三层内容共享同一份：

- 研究数据
- 来源体系
- 素材候选池
- 核对日期体系

### 2. Intake 策略

请求采集采用“两段式补问”：

1. 缺核心字段时暂停，先追问
2. 缺次级偏好时继续研究，并记录默认值或未明确状态

核心字段包括：

- 出发城市
- 目的地与途经城市
- 出行日期范围
- 出行人数
- 是否有小孩
- 年龄段概况
- 预算范围

次级偏好包括：

- 高铁 / 飞机 / 自驾偏好
- 住宿偏好
- 节奏偏好
- 是否接受早起、转车、换城市
- 是否偏好拍照、美食、自然、亲子、轻松、密集行程
- 必去项与明确不去项

当存在小孩、老人或较大年龄跨度时，V2 需要把这类信息继续作用到后续内容生成：

- 行程节奏建议
- 步行强度建议
- 交通换乘复杂度建议
- 穿衣与必备物品提醒
- 餐饮口味与就餐时段安排

### 3. 高时效信息写法

V2 对以下信息默认不写成绝对确定事实：

- 具体班次
- 具体票价
- 精确未来天气
- 店铺当日营业状态
- 景区临时预约规则

正文默认写法为：

- 最近可查方案
- 价格区间
- 核对日期

更细的车次、航班、交通细表、截图来源等进入 JSON 和附表层，不在主正文中伪装成确定事实。

### 4. 研究数据落盘粒度

研究数据默认采用“地点 + 主题”双层归档，并同步产出总汇总文件：

```text
travel-run/
├─ intake/
├─ research/
│  ├─ raw/
│  │  ├─ <place>/<topic>/*.json
│  ├─ normalized/
│  │  ├─ <place>/<topic>.json
│  ├─ summary/
│  │  ├─ transport-summary.json
│  │  ├─ weather-summary.json
│  │  ├─ food-summary.json
│  │  ├─ attractions-summary.json
│  │  └─ risk-summary.json
│  └─ research-summary.json
├─ review/
├─ compose/
├─ render/
├─ dist/
└─ verify/
```

这样可以支持：

1. 补某个地点的资料
2. 补某个主题的资料
3. 用同一研究底座生成不同层级攻略
4. 后续做插图候选和风格分支

### 5. 视觉策略

V2 默认只输出一套正式样式，但必须保留主题接口。

含义是：

1. 主流程优先保证信息完整、结构稳定、双端一致
2. 样式主题作为扩展层接入 `frontend-design` 和 `theme-factory`
3. 不要求每次默认同时产出多风格稿，但要允许后续按客户需要分支生成

## 内容结构规范

### 总章节顺序

三层攻略共用同一套大章节顺序，只是信息密度不同：

1. 行程总览与每日安排
2. 穿衣、天气、温度、必备物品
3. 最推荐路线与交通方式
4. 最全面交通与备选方式
5. 景点详细介绍
6. 美食与店铺推荐
7. 当前月份可看 / 不可看内容
8. 最佳季节建议
9. 注意事项
10. 信息来源

### 每日行程速览

`daily-overview` 固定包含：

1. 行程摘要
2. 每日安排表
3. 当天主交通
4. 当天重点景点
5. 当天推荐餐饮
6. 当天穿衣提醒
7. 当天必备物品提醒
8. 当天注意事项
9. 来源简表

### 最推荐攻略

`recommended` 固定包含：

1. 行程总览与适合人群
2. 穿衣、天气、温度、必备物品
3. 最推荐路线与交通方式
4. 每日详细安排
5. 景点详细介绍
6. 推荐美食店铺
7. 当前月份能看到什么
8. 最佳出行月份与季节建议
9. 注意事项
10. 信息来源

### 最全面攻略

`comprehensive` 固定包含：

1. 行程总览与路线逻辑
2. 每日安排总表
3. 全部交通方案
4. 全部景点清单
5. 全部餐饮清单
6. 住宿区域与落脚建议
7. 当前月份与最佳季节对比
8. 穿衣与必备物品完整版
9. 风险与注意事项完整版
10. 全量来源与核对记录

### 景点字段

景点必须具备：

- 名称
- 所在城市 / 区域
- 简介
- 推荐理由
- 适合停留时长
- 门票 / 价格区间
- 预约方式
- 预约时间建议
- 当前月份可看内容
- 当前月份较难看到的内容
- 最佳观赏月份
- 来源与核对日期

### 店铺字段

店铺必须具备：

- 店名
- 城市 / 区域
- 详细地址
- 菜系
- 推荐菜
- 人均价格区间
- 营业时段
- 适合安排在哪一餐
- 风格标签
- 来源平台
- 来源与核对日期

### 交通字段

交通方案必须具备：

- 出发地
- 到达地
- 交通类型
- 适用层级
- 最近可查班次或最近可查方案说明
- 预计总时长
- 预计价格区间
- 中转或换乘说明
- 市内接驳方式
- 是否适合老人 / 小孩同行
- 核对日期
- 来源与备注

其中交通类型至少覆盖可查到的：

- 飞机
- 火车 / 高铁
- 地铁
- 公交
- 打车
- 自驾
- 步行接驳

`recommended` 层保留最合适主方案与少量备选，`comprehensive` 层需要完整展开多种交通路径、费用区间和适用场景。

## 双端与分享规范

### 目录结构

每个旅行项目默认输出：

```text
trips/<slug>/
├─ desktop/
│  ├─ daily-overview/
│  │  └─ index.html
│  ├─ recommended/
│  │  └─ index.html
│  └─ comprehensive/
│     └─ index.html
├─ mobile/
│  ├─ daily-overview/
│  │  └─ index.html
│  ├─ recommended/
│  │  └─ index.html
│  └─ comprehensive/
│     └─ index.html
├─ assets/
├─ notes/
└─ data/
```

### 桌面端规则

桌面端采用长页滚动，适合：

- 全局导航
- 更高密度表格
- 左右分栏
- 店铺与交通清单对照
- 多层导航切换

### 手机端规则

手机端内容必须与桌面端一致，但采用分页组织：

- 先展示每日安排
- 再按主题分页
- 每页只承载一个主要阅读目标
- 允许更细粒度拆页，不能靠删内容保持轻量

### 单文件分享策略

V2 默认输出 4 类分享物：

1. `portal.html`
   - 单文件入口页
   - 可导航到三层攻略与桌面端 / 手机端说明
2. `recommended.html`
   - 单文件最推荐攻略
3. `comprehensive.html`
   - 单文件最全面攻略
4. `zip`
   - 包含入口页、单文件攻略、来源清单、trip summary、可选 PDF

### 分享优先级

默认分享优先级为：

1. 单文件 HTML
2. ZIP
3. 静态 URL

原因：

- 单文件最稳
- ZIP 便于转发和归档
- 公开 URL 依赖部署能力，不作为默认前提

## 联网素材与媒体规范

### 默认素材来源

联网研究需要支持：

- 官网
- 12306
- 航司 / 机场
- 景区预约页
- 小红书
- 抖音
- B站
- 美团
- 大众点评

### 媒体数据落盘

素材默认按结构化方式落盘，而不是只留下链接：

- 原帖 / 原视频链接
- 平台
- 标题
- 作者
- 发布时间
- 摘要
- 评论区高价值信息
- 语音转文字结果
- 视频时间轴要点
- 候选画面时间点
- 插图用途建议

### 媒体产物策略

默认不直接把第三方平台素材作为最终成品主图，而是先生成：

- 图片候选清单
- 插图建议清单
- 视频画面建议时间点
- 封面建议
- 每页配图建议

## 文案与语气规范

正文必须面向读者，不暴露生成过程。

### 固定要求

1. 不写模型思考过程
2. 不写“参考你给的配色方向”之类的过程说明
3. 不写“这版把答案放前面”这类创作自述
4. 不把设计推理写进读者正文

### 语气要求

正文统一采用较温和表达：

- 用“最合适、比较合适、这样安排会更省力、更省事”
- 注意事项用正向提醒
- 风险提示用“要注意、建议提前核对、这样处理会更稳妥”

### 避免用语

以下类型表达不作为默认正文风格：

- 过强否定句堆叠
- 生硬判断词
- 明显像内部创作旁白的句子

## 流程编排规范

V2 固定采用 8 个阶段：

1. `intake`
2. `research-plan`
3. `research-run`
4. `review-gate`
5. `compose`
6. `render`
7. `package-share`
8. `verify`

### 阶段说明

#### 1. intake

输出：

- `trip-request.json`
- `missing-fields.json`

#### 2. research-plan

输出：

- `research-plan.json`

研究任务按三维拆分：

- 地点
- 主题
- 平台

主题默认至少覆盖：

- 天气与温度
- 穿衣建议
- 必备物品
- 大交通
- 市内交通
- 景点
- 门票与预约
- 美食店铺
- 住宿落脚区
- 当前月份可看内容
- 最佳季节
- 注意事项
- 信息来源与核对记录

#### 3. research-run

输出：

- `research/raw/...`
- `research/normalized/...`
- `research/research-summary.json`

#### 4. review-gate

输出：

- `review/research-packet.md`
- `review/research-packet.html`
- `review/research-packet.json`
- `review/approved-research.json`

这是强制人工确认闸门。

#### 5. compose

输出：

- `compose/content-model.daily-overview.json`
- `compose/content-model.recommended.json`
- `compose/content-model.comprehensive.json`

#### 6. render

输出：

- 桌面端三层页面
- 手机端三层页面
- 共享 assets / notes / data

#### 7. package-share

输出：

- `portal.html`
- `recommended.html`
- `comprehensive.html`
- `zip`
- 可选 `pdf`

#### 8. verify

输出：

- `verify/report.json`
- 可选 `verify/report.md`

## Skill 协同规范

`travel-skill` 自身负责编排和合同，不直接承担所有专项能力。

### 强依赖 skill

1. `web-access`
   - 官网、交通、天气、社媒、餐饮平台数据采集
2. `frontend-design`
   - 桌面端和手机端成品页面设计
3. `playwright-skill` 或现有 Playwright checker
   - 页面渲染与交互校验

### 可选依赖 skill

1. `theme-factory`
   - 主题分支与风格扩展
2. `visual-explainer`
   - 把 review-gate 或流程输出做成可视化说明页
3. `last30days`
   - 近期趋势补充

V2 需要保留现有脚本，并补足以下能力：

### 已有核心脚本

- `normalize_request.py`
- `build_research_tasks.py`
- `merge_sources.py`
- `extract_structured_facts.py`
- `generate_review_packet.py`
- `build_guide_model.py`
- `fill_missing_sections.py`
- `render_trip_site.py`
- `export_single_html.py`
- `package_trip.py`
- `verify_trip.py`

### 建议新增脚本

- `build_portal.py`
- `collect_media_candidates.py`
- `build_image_plan.py`

### 新增脚本职责

#### `build_portal.py`

负责生成单文件入口页，统一导航：

- 每日行程速览
- 最推荐攻略
- 最全面攻略
- 桌面端 / 手机端说明

#### `collect_media_candidates.py`

负责把社媒正文、评论、视频时间轴、候选画面整理成结构化素材 JSON。

#### `build_image_plan.py`

负责为各页面区块生成：

- 插图建议
- 图位表
- 封面建议
- 每页配图建议

## 数据合同

V2 的核心对象固定为：

1. `trip_request`
2. `research_task`
3. `research_note`
4. `fact_item`
5. `content_item`
6. `source_item`

这些对象是脚本层、渲染层、导出层和验证层共享的正式合同，不再把 JSON 只当中间缓存。

## 失败与回退策略

### 缺核心字段

停在 `intake`，先追问。

### 缺次级偏好

继续研究，并显式标记默认偏好或未明确状态。

### 查不到精确班次

正文写最近可查方案、价格区间、核对日期，详细明细进入附表和 JSON。

### 查不到店铺稳定营业信息

允许保留店铺，但标记建议临近出发复核。

### 图片素材不稳定

输出图片候选清单，不强行进入主页面。

### 手机端分页过长

自动进一步拆页，不靠删内容缩短。

## 样例定位

`jilin-yanji-changbaishan` 是 V2 的代表性验证样例，原因是它天然具备：

- 多城市切换
- 多交通方式
- 山区与城市体感并存
- 景点、餐饮、社媒信息密度高
- 需要兼顾执行层信息与阅读层体验

V2 规范要能覆盖它，但不能写成只服务这一条路线的专用规范。

## 风险与后续注意事项

1. 社媒平台结构和可抓取性会波动，`web-access` 结果必须保留来源与核对时间。
2. 单文件 HTML 如果直接内联大量图片，体积会偏大，因此默认把图片作为候选池而不是全部硬塞进页面。
3. 若后续扩展到更广范围的目的地，需要继续维护地点标签、主题标签和写作规则，但 V2 的对象合同和流程结构不应被打散。

## 决策总结

本设计文档的最终结论是：

1. V2 面向国内多城市、多交通方式、强信息密度旅行场景。
2. 默认产物固定为三层攻略，每层同时输出桌面端和手机端。
