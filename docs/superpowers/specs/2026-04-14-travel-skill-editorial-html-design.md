# travel-skill 吸收 frontend-design 的高端旅游杂志式 HTML 设计方案

日期：2026-04-14  
状态：draft  
范围：仅限 skill 设计与文档重构，不包含渲染脚本或模板代码改造

## 1. 背景

当前 `travel-skill` 已覆盖旅游攻略的 intake、research、planning、compose、render、package、verify 主流程，但最终 HTML 产物仍带有明显模板导向。用户希望：

- `travel-skill` 单独触发时即可完成高质量旅游攻略 HTML 设计；
- 不再依赖额外显式调用 `frontend-design`；
- 旅游 HTML 的内容顺序继续遵循 `.codex/skills/travel-skill` 既有规则；
- 现有多套模板思路先被摒弃，不再作为默认输出目标。

本次设计只处理 skill 层能力吸收与规范重写，不直接修改 render 脚本、模板实现或验证脚本。

## 2. 目标

改造后的 `travel-skill` 应满足以下目标：

1. **独立设计能力**：单独触发时即可指导生成高质量 HTML 旅游攻略，不要求显式并行调用 `frontend-design`。
2. **内容契约不变**：继续遵守 `daily-overview`、`recommended`、`comprehensive` 的既定内容模型与章节顺序。
3. **默认审美基线固定**：采用“高端旅游杂志式”作为默认 HTML 风格基线，而非按任务随意漂移。
4. **旧模板退场**：现有“固定多模板批量输出”的思路降级为非默认、待废弃资产。
5. **可扩展**：未来如需恢复多风格输出，可在不破坏现有内容契约前提下继续扩展。

## 3. 非目标

本轮明确不做以下事项：

- 不重写现有 HTML 模板文件；
- 不改写 render pipeline 的实现代码；
- 不新增图片生成、地图绘制、动画脚本等实现细节；
- 不改变 `travel-skill` 的研究流程、数据落盘路径和来源验证规则；
- 不改变 guide 内容顺序。

## 4. 设计原则

### 4.1 内容优先，设计增强

`travel-skill` 的核心仍是研究与内容组织。视觉设计负责增强阅读体验、建立叙事节奏、提升分享价值，但不能篡改内容结构。

### 4.2 顺序稳定，版式自由

发布 HTML 时，章节顺序必须遵守 `travel-skill` 原有规则；允许在视觉层使用封面、章节过渡、图文编排、强调块、引文块、行程时间轴等方式增强表现，但不能重排 section 顺序。

### 4.3 默认风格固定

默认风格统一为：

- 高端旅游杂志式（editorial / refined / immersive）
- 强层级标题系统
- 大幅留白与节奏控制
- 图文混排而非卡片堆砌
- 类纸媒阅读感而非 SaaS 控制台感

### 4.4 明确反模板化

禁止把旅游攻略页面退化成：

- 通用 dashboard
- SaaS 落地页
- 资讯站列表页
- 紫渐变 AI 风
- 无叙事层次的卡片宫格

## 5. 方案对比

### 方案 A：仅在 `SKILL.md` 中补几条设计要求

优点：
- 修改快；
- 成本最低。

缺点：
- 设计约束太弱；
- 后续容易再次退化为普通模板页；
- 很难支撑“单独触发即可独立设计”的目标。

### 方案 B：重写 `SKILL.md` 并补 travel-skill 自有设计参考文档（推荐）

优点：
- `travel-skill` 可独立完成设计决策；
- 不必机械复制 `frontend-design` 全文；
- 可保持设计规范与旅游业务规则紧耦合；
- 后续实现脚本时有明确依据。

缺点：
- 文档层次略增；
- 本轮仍不会直接带来渲染代码升级。

### 方案 C：文档与 render/verify 一起重构

优点：
- 一步到位；
- 能立即把规范落实到产物。

缺点：
- 范围超出本轮；
- 风险更高；
- 不符合当前“先做 A”的要求。

结论：采用**方案 B**，以文档吸收为主，保留未来进入 C 的空间。

## 6. 目标结构

### 6.1 `SKILL.md` 重写方向

`travel-skill/SKILL.md` 需要做以下调整：

1. 删除“最终阅读体验依赖 `frontend-design`”的外部协调表述；
2. 增加 `travel-skill` 自身的 HTML 设计职责声明；
3. 将默认审美方向写死为“高端旅游杂志式”；
4. 在 render 规则中删除“固定五模板输出”导向；
5. 增加设计决策前置步骤与视觉质量门槛；
6. 保留原有章节顺序、研究规则、验证原则。

### 6.2 新增参考文档

建议新增以下文件：

1. `references/html-design-baseline.md`
   - 定义默认风格基线；
   - 说明字体气质、留白原则、图文关系、节奏控制、色彩策略。

2. `references/html-visual-rules.md`
   - 规定封面、章节页头、段落栅格、时间轴、交通信息块、tips/alert/source 的视觉模式；
   - 规定移动端与桌面端如何保持同事实、不同节奏。

3. `references/html-anti-patterns.md`
   - 定义廉价感来源；
   - 列出明确禁止的布局、组件感、配色和交互套路。

### 6.3 对现有模板资产的处理

现有模板文件本轮不删除，但在文档层明确：

- 它们不是默认输出目标；
- 它们不再代表 `travel-skill` 的设计方向；
- 后续如继续存在，应视为迁移前遗留资产。

## 7. HTML 设计基线

### 7.1 默认视觉定位

默认视觉定位为“高端旅游杂志式”：

- 像一本可分享的旅行特刊，而不是工具页；
- 首页强调旅行主题、气候氛围、目的地气质和路线承诺；
- 章节切换强调叙事节奏而不是模块堆叠；
- 信息密度高，但阅读压迫感低。

### 7.2 版式原则

- 以大标题、导语、章节引入形成阅读节奏；
- 通过跨栏、图文穿插、分组留白拉开层次；
- 列表和表格只在确有必要时使用；
- 不以统一卡片外框统治整页；
- 每个 section 都应有清晰的进入点、信息主体、收束点。

### 7.3 信息表现原则

针对 `travel-skill` 内容顺序，要求：

- `recommended_route`：作为主叙事段落，允许使用英雄区、路线摘要、时间轴；
- `route_options`：作为比较段落，强调差异与决策；
- `clothing_guide`：强调温度、层次、场景建议，避免电商式块状推荐；
- `attractions`：强调场景感、节奏、避坑，而非景点卡片瀑布流；
- `transport_details`：强调可执行信息与时间/价格层级；
- `food_by_city`：允许更生活方式化，但仍需受主风格统一；
- `tips`：适合警示块、经验块、边注样式；
- `sources`：清晰但克制，不能喧宾夺主。

### 7.4 设计禁令

明确禁止：

- 常见紫粉蓝渐变科技风；
- 通用玻璃拟态乱用；
- 全页等宽卡片网格；
- 每段都包裹边框；
- 过度按钮化、标签化；
- 头图之后直接进入“八宫格功能区”；
- 与旅游内容无关的 SaaS icon 风信息块。

## 8. 渲染与验证规则的文档改写方向

虽然本轮不改 render/verify 代码，但文档需提前改成新的目标约束：

### 8.1 新的 render 表述

将原先“固定五模板产出”改写为：

- 默认产出一套统一风格的 desktop/mobile HTML；
- 风格默认采用高端旅游杂志式；
- 允许未来扩展为多风格，但本 skill 当前默认不要求多模板批量发布；
- 内容顺序严格服从 guide contract。

### 8.2 新的 verify 表述

在原有完整性验证之外，新增文档级质量要求：

- 页面具备明确视觉层级；
- 章节组织具有杂志式阅读节奏；
- desktop/mobile 保持事实一致；
- `sources` 与 `notes` 保留；
- 页面不应退化为旧模板式卡片集合。

## 9. 实施边界

本次实施预计只涉及：

- 修改 `.codex/skills/travel-skill/SKILL.md`
- 新增 2~3 份 `references/*.md`
- 对旧模板相关文档措辞做降级或废弃说明（如有必要）

不触碰：

- `scripts/`
- `travel-data/`
- render 脚本
- 测试脚本

## 10. 风险与缓解

### 风险 1：文档升级但产物暂时没变

原因：本轮不改实现层。  
缓解：在文档中明确这是“skill 能力定义先行”，后续 render 实现需按新规范收敛。

### 风险 2：规范过宽，仍可能生成普通页面

原因：若设计规则写得抽象，后续执行时会滑回模板化输出。  
缓解：增加 anti-pattern 文档与禁止项，用负面约束钉死下限。

### 风险 3：误伤现有模板链路

原因：若直接删模板，可能影响现有脚本。  
缓解：本轮不删资产，只做文档层退场声明。

## 11. 验收标准

当本方案实施完成时，应满足：

1. `travel-skill` 文档不再要求显式调用 `frontend-design`；
2. `travel-skill` 自身已经定义默认 HTML 审美基线；
3. 文档明确保留原内容顺序；
4. 文档明确放弃固定多模板默认输出；
5. 文档中存在明确的设计正向规则与反模式约束；
6. 后续执行者仅阅读 `travel-skill` 自身文档，也能知道怎样做出高质量旅游攻略 HTML。

## 12. 后续阶段建议

本设计完成并获批后，下一阶段应进入 implementation planning：

1. 具体改写 `SKILL.md` 哪些段落；
2. 如何组织新增 references；
3. 是否需要给旧模板加“deprecated”标识；
4. 是否补最小验证，检查文档中已不再要求五模板输出。
