# 旅行指南执行手册

本手册把旅行指南制作过程拆成可复用的执行步骤，确保每次接到新需求时都能迅速应对。把调研、生成、HTML 校验与图位计划看成流水线的不同车间，每个车间在它负责的阶段完成交接验收。

## 开始前必须确认的信息

- 需求联系人是谁？他/她的信号（渠道、时间窗口、预算）是否确定。
- 旅行日期和备用窗口（比如备选周末或延后日）。如果有硬性出发或返程航班，应标记出来。
- 参与者人数、身份、敏感需求（儿童、老人、团建、身体限制）及住宿偏好。
- 预算范围、支付方与核销节奏，是否有价格雷达或采购要求。
- 需要对接的合作方（行政、财务、品牌、合作酒店）和审批节点。
- 现成内容：已有的路线草稿、已选航班/酒店、固定品牌资产等，可复用的文字/图像。
- 内容格式需求：HTML 结构限制、语言偏好、是否必须含清晰导航和概览。
- 成功指标：对外交付物（桌面页、移动页、source note、image plan）与内部追踪（计划号、Jira/Notion 链接）。
- 风险预案：关键节点（票、酒店预订）出现问题时的应对路径。

## 调研执行顺序

1. 明确本次调研的输出标准：目标受众、交付格式、完成时限，以及和原始需求的差异。
2. 归档当前需求和约束：谁要去、什么时候去、借助哪些联动渠道发布、预算边界如何。
3. 收集基本事实：交通节点、热门景点、季节性气候、当地节庆、体验亮点、餐饮/住宿建议。
4. 把握日程节奏：预估行程节段（到达、沉浸、出发）并对应前后节奏与预留缓冲。
5. 梳理料想的模块顺序：概览、推荐计划、备选方案、景点、餐食、季节提示、打包/交通、来源等。
6. 整理外部资源：地图、官方公告、历史数据、价格参考，并记录获取时间、负责人与链接。
7. 确认审批流与联络人：谁最终签字、谁负责推送到运营/设计、谁能解决跨团队问题。
8. 核对体验语境：比如「长白山春游」和「延吉自驾」在风格和节奏上的落差，及时调整语气。
9. 抽出图位/素材需求：每个核心段落需要几张图、图名、取景节点、授权情况。
10. 规划沟通节奏：定期同步调研进展、关键发现，与需求方留出确认时间。
11. 复盘已知空白：对照手头的资料列出未解决的问题、预计的依赖和下一步行动项。

## 行程方案生成规则

在推荐方案和备选路线之间保持可比性，始终把「一天可实现」「体验一致」放在首位。

- 推荐方案要先说明承接哪种旅程心智，再说明主线路、核心体验、理想节奏与备用方案。
- 每个方案都要列出最容易实现的方向（比如「高铁＋延吉落地」），并说明结果好的前提条件。
- 对于后续方案，给出权衡项：时间成本、预算成本、活动强度、组合难度。
- 本文档鼓励分层：1) 推荐 24 小时安排 2) 备选「慢游」「沉浸」「小众」方案 3) 需要额外审批或高风险行为的提示。
- 所有方案都有明确的数据来源和可复用参考链接，让后续同事立即抓取素材。
- 交付前再次审核：是否留了编辑余地、是否标明必备信息（限时票、订房截止）。

## HTML 内容顺序规范

按固定顺序组织 HTML，便于桌面/移动并行：

1. 页眉/标题（含副本、日期范围、出发城市、目标人群）。
2. 短语眼（eyebrow）或子标题强调旅行主题。
3. 概览区（用一段 summary + 3-4 个高亮弹窗）。
4. 推荐方案（推荐 24 小时节奏、建议行程结构、节点与时间）。
5. 其他方案（至少 2 个选项：淡季/沉浸/亲子/小众）。
6. 景点与体验清单（卡片式、要素包含亮点 + 位置 + 必做）。
7. 餐食/休息/服务推荐（含价位提示、营业时间、预约方式）。
8. 季节与温差提示（气候、准备物、避风/预警信息）。
9. 打包/装备清单（衣物、证件、接口、预存物品）。
10. 交通/联程提醒（机场/高铁/包车安排、路线图、落地联络人）。
11. 信息来源与参考（列出官方/可靠来源、可引用链接）。

所有主内容块必须以 `<section id="...">` 包裹，`id` 只能使用 ASCII 小写字母与破折号（不允许空格或大写），且在页内唯一。将 HTML 结构按本节列出的顺序写出，以便 Playwright 与 renderer 一一映射，任何重复或跳号都会让后续自动校验失败。

| 中文标题 | 规范 `id` |
| --- | --- |
| 概览 | `overview` |
| 推荐方案 | `recommended` |
| 其他方案 | `options` |
| 景点/体验 | `attractions` |
| 餐食/休息 | `food` |
| 季节提示 | `season` |
| 打包/装备 | `packing` |
| 交通/联程 | `transport` |
| 来源与参考 | `sources` |

务必先写中文标题再贴 `id`，例如：

```
<section id="overview">
  <h2>概览</h2>
  ...
</section>
```

每一部分都要匹配 `id`（overview, recommended, options, attractions, food, season, packing, transport, sources）并按顺序命名。

## 图位方案规范

图位是该手册的视觉骨架：每个卡片都要交代图号、主题、场景范围、取景角度、交付规格与素材管理方式。

- 确定图位编号（A1、A2 等），并说明对应 HTML 区块。
- 指定每张图的场景设定：到达/欢迎、沉浸体验、反转景色、长白山/延吉地标。
- 说明拍摄或选图的触发条件：动态视角、黄金时间段、人物出现、长焦/广角比例。
- 指出图像使用渠道：桌面 hero、移动 page header、来源卡片、分享卡片，指导尺寸与 crop 注意事项。
- 列出现有素材（授权截图、公共图集）和缺口（需拍摄/购买的内容）。
- 维护图位负责人，对接摄影/设计，明确交付期限和版本号。
- 最后附加「如果无图」的占位策略：调色背景、图标 + 口号、关键数据条。

## 交付检查清单

- 确认文字稿：标题、摘要、推荐/备选方案、景点、交通等都有对应的结构化段落。
- 确认 HTML 结构与 `id` 是否齐全并按顺序布局。
- 核对图位表格：编号、场景、触发、渠道、负责人、交付状态。
- 资料来源：列出至少 3 个官方/本地/合作来源，并在 sources 部分标注。
- 图片准备：高光图片、备用图片、如需补拍/采购的清单。
- 交付前预览：桌面/移动样式都跑过一次（手动或 Playwright），记录主要差异。
- Playwright 验证流程：`python tests/playwright_trip_render_check.py`，检查桌面/移动页面在常见断点（约 1280px、768px、360px）下是否返回所有 `section id`、无横向溢出并且分页标签存在，将 `travel guide regression checks passed` 视为接收信号。
- 复用信息库：把本次调研的有用片段（数据、角度、素材）加入知识库。

## 可复用输出模板

保证每次交付自带「窗口感」与「场景感」，并能快速套入新内容。

### 核心信息卡

- 项目名称、出发城市、日期范围、参与人数与重点人群。
- 推荐路线概述（标题 + 1-2 句摘要 + 1 个高亮亮点）。
- 成本/时间等级（如：√ 预算 3-4k、√ 全天/2 天、√ 家庭友好）。
- 关键联系人（需求人、审批人、运营协调）。

### 输出清单

- 桌面 HTML + CSS（路径 `trips/<trip-slug>/desktop/index.html` + `desktop.css`，命名保持 `desktop` 目录并加载 `assets` 共享样式/脚本）。
- 移动 HTML + CSS（路径 `trips/<trip-slug>/mobile/index.html` + `mobile.css`，需设置 `data-page` 分页并与桌面共享 `assets`）。
- Shared content JS + renderer：`trips/<trip-slug>/assets/guide-content.js`（导出 `tripGuide`，`meta` 与 `sections` 按照规范 `overview` 等顺序）和 `trips/<trip-slug>/assets/render-guide.js`（暴露 `renderCards`、`renderSection`、`byId` 等辅助函数）。命名须保持 `guide-content.js` / `render-guide.js`，永远在 `assets` 目录下。
- Research/delivery notes：`trips/<trip-slug>/notes/sources.md`（按主题分组、注明来源/日期/联系人）、`trips/<trip-slug>/notes/image-plan.md`（包含图位表格）、`trips/<trip-slug>/notes/research-log.md`（或等效命名，须记录调研路径/日期/结论/待办）。头部需写明 slug、版本、审核人。

### 客户确认提示模板

```
[旅程标题]：完成
- 日期：______
- 关键节点：__________________
- 需要重点说明的风险/不确定性：______________________
- 交付物：桌面 / 移动 / 研究笔记 / 图位计划
```

如果需要快速复刻新旅程，可把上述卡片复制，替换核心信息即可；其它细节（图位、图像）再按本手册安排。
